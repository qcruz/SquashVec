# DED Red Team Scenarios

**Directory:** `ded/redteam/`
**Last Updated:** 2026-05
**Purpose:** Creative development and program stress-testing

---

## What This Is

Red team scenarios are semi-realistic, highly specific disruption events used to:

1. **Stress-test the DED program library** — force program combinations against unexpected inputs
2. **Identify gaps** — find scenarios that existing programs address poorly or not at all
3. **Refine cost/benefit analysis** — move beyond generic scenario tiers to specific event modeling
4. **Pressure-test assumptions** — see if the DED mandate philosophy holds up under unusual conditions

These are not the canonical DED scenarios (oil restriction, pandemic, etc.) — those are
categories. Red team scenarios are **specific events** with dates, casualty estimates, named
institutions, realistic market dynamics, and cascades that follow cause-and-effect logic.

The goal is to produce the most useful DED planning insight, not the most spectacular scenario.
A poorly-analyzed massive crisis is less useful than a tightly-analyzed modest one.

---

## Format

Each scenario document follows this structure:

1. **Situation** — the triggering event, with specific realistic detail
2. **Cascade Timeline** — how the event spreads across systems, day by day and month by month
3. **Economic Damage Estimate** — total expected loss by category with methodology
4. **DED Program Mapping** — which programs activate, at which phase, in what sequence
5. **Optimal Program Combination** — the specific mix that minimizes cost and maximizes benefit
6. **What the DED Wouldn't Have** — gaps the scenario exposes; programs or pre-builds missing
7. **Lessons for Pre-Positioning** — what should be built before this scenario occurs
